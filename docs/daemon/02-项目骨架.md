# 02 · 阶段 1 — 项目骨架

> 目标：在 `apps/daemon` 下建一个最小可启动的 Node.js + TypeScript 程序。**约束：能被 `pnpm dev:daemon` 跑起来、每秒打一行 log、Ctrl-C 在 100ms 内干净退出。** 故意不引入 HTTP / pino / zod / hono——这些留给后续阶段。

---

## 1. 为什么先搭一个「啥也不干」的骨架

后续阶段会一层层加：Config（2）→ Health HTTP（3）→ Backend（4）→ Task API（5）→ Electron 集成（6）→ 打包（7）。每一层都会**重写** main.ts 的装配部分。

如果不先有一个跑得起来的骨架：

- 阶段 2 加 pino 时，你得同时处理「pino 怎么配」+「Node 启动该怎么写」两个未知数，出错时分不清是哪一边的锅。
- 阶段 3 加 HTTP server 时，你需要一个已经能干净退出的进程来验证「HTTP `/shutdown` 是否真的触发了关闭」。

所以本阶段的策略是：**把所有复杂度砍到零，只验证两件事**——能启动、能优雅退出。这两件事是后续所有阶段的地基。

## 2. 四个核心概念

### 2.1 `apps/daemon` 作为 workspace 成员

仓库根的 `pnpm-workspace.yaml` 已经声明 `packages: [apps/*, packages/*]`，所以**新建 `apps/daemon/` 后它自动成为 workspace 成员**，不需要改 workspace 配置。包名 `@demo/daemon`，其他包可以用 `"@demo/daemon": "workspace:*"` 引用（虽然短期内没人会引用它）。

### 2.2 用 `tsx` 直接跑 TypeScript

正常 Node.js 不能直接跑 `.ts`——需要 `tsc` 编译或 `tsup` 打包。开发时这层编译会拖慢迭代。

**`tsx`** 是一个「Node.js 的 TypeScript 加载器」：它用 esbuild 在运行时转译 `.ts`，启动延迟 <100ms。用法：

```bash
tsx src/main.ts           # 跑一次
tsx watch src/main.ts     # 文件改动自动重启
```

`tsx watch` 是 daemon 开发时的主力——改代码不用手动重启。**prod 不用 tsx**（阶段 7 会用 `tsup` 打成单文件 `dist/daemon.js`，用 `node` 跑）。

> 对照 frontend：`apps/web` 用 Vite 的 HMR，`apps/desktop` 用 electron-vite 的 watch。daemon 不需要 HMR，文件变化整个进程重启足够——状态本来就都在内存里。

### 2.3 `AbortController` 作为单点关闭开关

这是本阶段最重要的概念。整个 daemon 后续会有多个并发组件：HTTP server、心跳定时器、poll 循环、正在跑的 task……优雅关闭时要让它们**同时**收到「该退出了」的信号。

**`AbortController` 是 Node 内置（Web 标准）的单点开关**：

```typescript
const controller = new AbortController();

// 派生：传给各组件
fetch(url, { signal: controller.signal });
setInterval(fn, 1000);  // 自己监听 signal

// 触发：一处 abort，所有持有 signal 的组件同时收到
controller.abort();
```

multica 用 Go 的 `context.CancelFunc` 做同样的事，TS 的对应物就是 `AbortController`。**语义等价**——这是 Go→TS 翻译里最干净的一对。

后续阶段会把 `controller.signal` 传给：
- 阶段 3 的 `httpServer.close()`（`/shutdown` handler 调 `controller.abort()`）
- 阶段 4 的 `spawn(...)` 子进程（`signal.addEventListener('abort', () => child.kill())`）
- 阶段 5 的所有正在跑的 task（`task.abortSignal = controller.signal`）

本阶段先把骨架搭对，不展开。

### 2.4 Node.js 的信号处理

操作系统发给进程的信号：

| 信号 | 来源 | 默认行为 |
|---|---|---|
| `SIGINT` | Ctrl-C | 退出（exit 130） |
| `SIGTERM` | `kill <pid>`、Docker stop、systemd | 退出（exit 143） |
| `SIGHUP` | 终端关闭 | 退出 |

如果**不**注册 handler，Node 默认直接死——没机会清理。daemon 需要清理（关 HTTP server、等 task 结束、flush 日志），所以要 `process.on('SIGINT', ...)` 拦截。

**Windows 坑**：

- `SIGINT` 在 Windows 下能可靠触发（Ctrl-C）。
- `SIGTERM` 在 Windows 下**不能**用 `kill` 命令发送——Windows 没有 POSIX 信号。但 Node 的 `process.on('SIGTERM')` 仍能捕获某些场景（如父进程调用 `subprocess.kill()`）。
- 真正在 Windows 下「杀进程」是 `taskkill /F /T /PID <pid>`——这会直接死，**没有**优雅退出的机会。

这就是为什么 multica 在 daemon 里**额外**起了 `/shutdown` HTTP 端点（对标 `server/internal/daemon/health.go` 的 `shutdownHandler`）——Windows 下 `multica daemon stop` 走 HTTP 而不是信号。本阶段先只接信号，阶段 3 加 `/shutdown` 把它和信号合并到同一个 `controller.abort()` 路径。

## 4. 关键文件内容

### 4.1 `apps/daemon/package.json`

```json
{
  "name": "@demo/daemon",
  "private": true,
  "type": "module",
  "scripts": {
    "dev": "tsx watch src/main.ts",
    "start": "tsx src/main.ts",
    "typecheck": "tsc --noEmit"
  },
  "devDependencies": {
    "@types/node": "^22.10.2",
    "tsx": "^4.19.2",
    "typescript": "^5.6.3"
  }
}
```

说明：

- `type: module`：Node 原生 ESM。`.ts` 里写 `import` 直接生效。
- `dev`：watch 模式，文件改动自动重启，开发主力。
- `start`：不带 watch，跑一次。`tsx` 仍然在运行时转译——**这是 dev/early-prod 用的**。阶段 7 会加 `build`（tsup 打成纯 JS）和真正的 prod 启动脚本。
- `typecheck`：`tsc --noEmit` 只检查，不产出。
- **故意不写 `build`**：阶段 7 才需要。先不占位。

> `tsx` 和 `typescript` 的版本写 `^4.19.2` / `^5.6.3`——比较保守的最新 major。不用 `catalog:` 因为 daemon 不和前端栈共享这些。

### 4.2 `apps/daemon/tsconfig.json`

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["ES2022"],
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true,
    "types": ["node"],
    "outDir": "./dist",
    "noEmit": true
  },
  "include": ["src/**/*"]
}
```

说明：

- `target: ES2022` + `lib: ["ES2022"]`：Node 20+ 全支持，Top-level await、`AbortController`、`structuredClone` 等原生可用。
- `moduleResolution: Bundler`：tsx 用 esbuild，走 Bundler 模式最顺。
- `strict: true` + `noUncheckedIndexedAccess: true`：对齐项目里其他 tsconfig 的严格度。
- `verbatimModuleSyntax: true`：强制 `import type` 分离类型与值导入——避免 tsx 在转译时把「只用做类型」的 import 错误地保留。
- `types: ["node"]`：只引入 `@types/node`，避免被其他包的 `@types/*` 污染全局。
- `noEmit: true`：本 tsconfig 只做类型检查。实际运行靠 tsx，打包靠阶段 7 的 tsup。

### 4.3 `apps/daemon/src/main.ts`

```typescript
// daemon 入口 · 阶段 1：最小骨架
// 故意不引 HTTP / 日志库 / 配置库；这些从阶段 2 起逐层加入。

// 单点关闭开关。所有后续组件（HTTP server、子进程、task runner）
// 都会收到同一个 controller.signal；调用 controller.abort() 后，
// 它们在下一次 await / 事件循环时收到取消信号，各自清理退出。
const controller = new AbortController();

let shuttingDown = false;
function shutdown(reason: string): void {
  if (shuttingDown) return; // 重复信号（Ctrl-C 连按两次）只处理一次
  shuttingDown = true;
  console.log(`[daemon] shutdown triggered: ${reason}`);
  controller.abort(new ShutdownReason(reason));
}

// 自定义 abort reason，便于后续组件区分「正常关闭」和「异常中断」
class ShutdownReason extends Error {
  constructor(reason: string) {
    super(reason);
    this.name = "ShutdownReason";
  }
}

// Windows 下 SIGTERM 不可靠（Windows 没有 POSIX 信号），
// 但 SIGINT（Ctrl-C）总能触发。两者都注册，覆盖各平台。
process.on("SIGINT", () => shutdown("SIGINT"));
process.on("SIGTERM", () => shutdown("SIGTERM"));

// 主循环：每秒打一行 log，证明 daemon 还活着。
// 阶段 6 之后这个 interval 会被删，换成真正的「主循环」装配。
const tick = setInterval(() => {
  console.log(`[daemon] alive pid=${process.pid} ts=${new Date().toISOString()}`);
}, 1000);

// abort 后的清理：停 tick、给 stdout 一点 flush 时间、exit。
controller.signal.addEventListener("abort", () => {
  clearInterval(tick);
  console.log("[daemon] bye");
  setTimeout(() => process.exit(0), 50);
});

console.log(`[daemon] started pid=${process.pid}`);
```

几个要点：

- **`shuttingDown` 守卫**：用户狂按 Ctrl-C 时只触发一次清理。
- **`ShutdownReason`**：后续组件如果需要区分「被 abort 了」和「`fetch` 自己 timeout 了」，可以 `signal.reason instanceof ShutdownReason` 判断。本阶段还没用到，但架子先搭这。
- **`setTimeout(..., 50)`**：stdout 是异步的，`process.exit(0)` 后没 flush 的 log 会丢。给 50ms flush。后续阶段换成 pino 的 `await logger.flush()` 或 `process.stdout.write` 的 callback。
- **`process.exit(0)` 是必需的**：如果不显式 exit，interval 已清 + 没有其他事件源，Node 会自然退出，**但** abort 后如果有未关闭的 handle（比如阶段 3 会有的 `httpServer`），Node 不会自动退。养成显式 exit 的习惯。

## 6. 常见陷阱

### 6.1 Ctrl-C 后不退出

**症状**：按 Ctrl-C 看到 `[daemon] shutdown triggered: SIGINT` 和 `[daemon] bye`，但进程不退。

**根因**：某个 handle 没关——通常是 `setInterval` 忘了 `clearInterval`，或者后续阶段 HTTP server 没 `server.close()`。Node 看到「还有待处理的工作」就不会自然退出。

**解法**：本阶段代码里显式 `clearInterval(tick)` + `process.exit(0)`。后续阶段每个组件都要在 `signal.addEventListener('abort', ...)` 里关掉自己的 handle。

### 6.2 `tsx watch` 改代码后旧进程没退干净

**症状**：改代码后日志里看到两个 pid 交替出现。

**根因**：tsx watch 重启时会发 SIGTERM，如果 handler 没正确 abort，旧进程可能残留几百毫秒。

**解法**：本阶段代码已经处理了 SIGTERM，应该没事。如果出问题，加一行 `console.log` 打印退出时间戳，看是不是卡在 `setTimeout` 那里。

### 6.3 Windows 下 `process.on('SIGTERM')` 不触发

**症状**：在 PowerShell 里 `Stop-Process -Id <pid>` 直接杀掉，没看到优雅退出日志。

**根因**：Windows 的 `Stop-Process` / `taskkill /F` 是强制杀，没有信号机制，进程没机会清理。

**解法**：这就是为什么阶段 3 要加 `/shutdown` HTTP 端点——Windows 下唯一可靠的优雅停服路径。**本阶段在 Windows 上验证时，只用 Ctrl-C**，不要用 Stop-Process。

### 6.4 ESM 导入忘加扩展名

**症状**：`import { foo } from './foo'` 报错 `ERR_MODULE_NOT_FOUND`。

**根因**：Node ESM 要求显式扩展名（`.js` / `.ts`）。tsx 放宽了这个限制，但**运行时还是按 Node 规则**——某些情况下 tsx 也救不了。

**解法**：写 `import { foo } from './foo.ts'`。后续阶段会频繁出现，提前适应。`moduleResolution: Bundler` 让 `tsc` 接受无扩展名，但运行时还是要带。
