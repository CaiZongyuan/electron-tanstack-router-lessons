# 01 · 架构与思维模型（阶段 0）

> 本文是动手前的理论基础。读完它，你应当能回答：「我们这个 daemon 到底在干什么？它和 multica 的 daemon 有什么本质差异？为什么 daemon 是一个独立进程而不是 Electron 主进程里的一段代码？」

---

## 1. 我们要做的不是 multica

multica 是一个 **cloud-orchestrated** 的多机 agent 网络：用户在 web 创建 issue，server 派发给某台机器的 daemon，daemon 在本地跑 claude，结果回传 server。

**我们要做的是「单机本地工具」**：用户打开 desktop 或 web 壳 → 在共享 UI 写 prompt → 本地 daemon 调 `claude` → 流式把输出显示到 UI。整个过程不出本机，**没有云端 task 分发**。

| | multica | 本项目 |
|---|---|---|
| Server 角色 | 中枢：task 分发、多机协同、状态管理 | 只做用户认证 |
| Daemon 角色 | 替 server 在本地跑 agent | 替本机 shell（desktop / web）在本地跑 agent |
| Task 来源 | 用户在 web 创建 issue → server 派发 → daemon poll | 用户在 desktop / web UI 直接发起 |
| Runtime 注册 | Daemon → Server（server 记账） | Daemon 探测到 claude 后直接暴露给本机 shell |
| 多机器 | 支持 | 不需要 |
| 凭证持有 | Daemon 持 server PAT | Daemon 不持任何凭证；claude CLI 自己管 token |

**为什么还要叫「daemon」**？因为它继承了 multica daemon 的三个架构精髓：

1. **独立进程**（不是 Electron 主进程的一段代码）
2. **本地 HTTP 健康端口 + 优雅关闭协议**
3. **Backend 抽象**（spawn agent CLI + 流式解析 stream-json）

学完这套骨架，你既理解了 multica daemon 的设计，也做出了一个有用的本地工具。未来如果想加「cloud 模式」（接真实 multica server 或自己的 hono server 派发 task），骨架已经就位，只是再加一层 client。

---

## 2. 为什么 daemon 独立进程，而不是塞进 Electron 主进程？

这是整个架构里**最关键的判断**。一旦理解它，后面所有的设计（health 端口、`/shutdown`、dev/prod 二分、IPC 模式）都是它的推论。

### 2.1 三条独立的「崩溃域」

```
┌─────────────────────────────────────────────────────────┐
│  Electron 主进程（app life）                            │
│   │                                                      │
│   ├── 渲染进程 ×N（用户 UI）                             │
│   │                                                      │
│   └── spawn ──▶ daemon 子进程（agent runtime）           │
│                   │                                      │
│                   └── spawn ──▶ claude code CLI          │
└─────────────────────────────────────────────────────────┘
```

**崩溃隔离**：`claude` 卡死、OOM、`V8 abort`、子进程 `SIGKILL`——任何一种都不该把 Electron 主进程一起拖死。daemon 独立进程后，agent 崩溃只杀 daemon，用户的窗口、未保存的设置、其他功能都不受影响。**对单机工具，这条价值最大**——用户体验最差的就是「写 chat 把整个 app 搞崩」。

**独立升级**：agent 后端迭代频繁（支持新 CLI、修 stream 解析 bug、加新 prompt 格式）。Electron 壳子则极其稳定。两者解耦后，未来可以单独升级 daemon（比如随 `claude` 新版推一个兼容 patch），不用重打整个 desktop 安装包。

**可独立调试**：daemon 是独立 Node 进程，可以用 `node --inspect` 附加、可以在终端前台跑、可以手动 curl 它的 HTTP API。塞进主进程就失去这些。

### 2.2 「Electron 被关 ≠ daemon 必须立刻死」

这是一个有用的副作用：daemon 独立后，Electron 卡死或被强制 kill 时，daemon 能自己检测父进程消失并优雅退出（或者根据配置继续跑完当前 task）。我们至少要用它保证**Electron 崩溃时 daemon 不孤儿**。

### 2.3 反面论证：什么时候不要独立进程

- **功能本身没有独立运行的可能**：比如「点击按钮打开外链」，这种同步链路不该做 daemon。
- **崩溃不会污染主进程**：纯内存的状态、只读 API，直接在主进程跑更简单。
- **启动延迟比隔离更重要**：daemon 冷启动要几百 ms（起 HTTP server、探测 claude）。

`claude` 执行符合「崩溃会污染 + 需要独立调试 + 流式 IO 重」三条，所以独立。

## 3. 数据流：一条 task 的完整旅程

共享的 `packages/views` chat UI 只认一个 `DaemonClient` 接口。用户在 UI 写「hello」到看到 claude 回复，shell 侧有两条接线，但下游 daemon API 完全一致：

```
┌─ packages/views 共享 chat UI ──────────────────────────┐
│  用户在 chat 输入「hello」，点发送                      │
│  ↓ daemonClient.runTask({ prompt: 'hello' })          │
└───────────────────────────────────────────────────────┘
                ↓
┌─ desktop 壳子 ─────────────────────────────────────────┐
│  renderer                                             │
│   ↓ IPC: 'daemon:runTask' / 'daemon:taskEvent'        │
│  preload                                              │
│   ↓                                                   │
│  main / daemon-manager                                │
│   fetch('http://127.0.0.1:19514/task/run', ...)       │
│   fetch('http://127.0.0.1:19514/task/t1/events')      │
└───────────────────────────────────────────────────────┘
                ↓ 或者
┌─ web 壳子 ─────────────────────────────────────────────┐
│  browser HTTP adapter                                 │
│   fetch('http://127.0.0.1:19514/task/run', ...)       │
│   fetch('http://127.0.0.1:19514/task/t1/events')      │
└───────────────────────────────────────────────────────┘
                ↓ HTTP localhost:19514
┌─ daemon ──────────────────────────────────────────────┐
│  /task/run handler:                                   │
│   - 生成 task_id = 't1'                               │
│   - 存到 taskStore                                    │
│   - 调 runner.run(task)  // 异步                       │
│   - 立即响应 { task_id }                               │
│                                                       │
│  /task/:id/events handler:                            │
│   - 设置 Content-Type: application/x-ndjson           │
│   - 先回放 taskStore 已有事件，再转 live 订阅           │
│   - 每条事件 JSON.stringify + 写 res + flush           │
│                                                       │
│  runner.run(task):                                    │
│   - backend = claudeBackend                           │
│   - for await (msg of backend.execute(prompt)) {      │
│       taskStore.publish(task.id, msg)                 │
│     }                                                 │
└───────────────────────────────────────────────────────┘
                ↓ spawn + stdin + stdout
┌─ claude code CLI ─────────────────────────────────────┐
│  $ claude -p --output-format stream-json --verbose    │
│  ◀── stdin: {"type":"user","content":"hello"}         │
│  ──▶ stdout: {"type":"assistant","content":"Hi!"}     │
│              {"type":"result","status":"completed"}   │
└───────────────────────────────────────────────────────┘
```

三个层次的关键设计：

- **共享视图只看 `DaemonClient`**：`packages/views` 不知道 IPC，也不写死 localhost URL。这样同一套 chat UI 才能同时跑在 desktop 与 web。
- **desktop main 是 lifecycle + 协议桥梁**：它把 IPC 消息翻译成 HTTP 请求，把 NDJSON 流翻译成 IPC 事件，并持有 daemon 子进程引用。
- **web adapter 直连同一套 HTTP API**：web 不负责拉起 daemon，只负责连接它；这样两端共用一套后端协议，而不是各搞一套 transport。
- **daemon 是协议终端**：从 HTTP 进，到子进程出，全程流式。

## 4. 关键设计决策

### 4.1 为什么 HTTP 推模式（task 不是 stdin）

web/desktop 壳子可以通过两种方式把 prompt 传给 daemon：

- **HTTP 推**（本项目）：daemon 起 19514 端口，desktop main 或 web browser `POST /task/run`。✅
- **stdin 推**：daemon 作为 desktop 子进程，stdin 写 JSON-RPC。❌

选 HTTP 的理由：

1. **同一套 API 供两种壳子复用**：desktop main 和 web browser 都能调同一个 `POST /task/run` / `GET /events`。
2. **多客户端共享**：未来 CLI 工具、第二个 Electron 窗口、调试脚本都能 HTTP 调 daemon，不需要 stdin pipe。
3. **调试方便**：`curl http://127.0.0.1:19514/task/run -d '{...}'` 就能试。
4. **`/health` 已经在 19514**：再多加几个路由几乎零成本。
5. **结构化错误**：HTTP status code + JSON body 比 stdin 的「自定义错误协议」标准。

stdin 推的好处是「不用占端口」，但 daemon 已经占 19514，这点好处消失；而且 web 壳子也没法复用 stdin 通道。

### 4.2 为什么 daemon 不持有 Claude 凭证

multica 的 daemon 持有 server PAT（Personal Access Token），用来调 server API。

**我们的 daemon 不持有任何凭证**：

- claude code CLI 自己管 token（用户在命令行 `claude login` 后存在 `~/.claude/`）。
- daemon 只负责 spawn `claude` 子进程，子进程自己读自己的配置。
- 你的 hono server 的用户认证由 Electron 主进程负责（用户登录存在 Electron 的 secure storage），daemon 完全不知道。

好处：

- **职责清晰**：daemon 是「transport」，不管 identity。
- **泄露面更小**：daemon 进程内存里没有敏感 token。
- **多用户友好**：未来如果同一台机器多个 OS 用户各跑各的 daemon，彼此隔离自然成立。

### 4.3 为什么 NDJSON 流式（不是 SSE / WebSocket）

daemon 把 claude 的流式消息推给 shell 有三种主流选项：

| 选项 | 优点 | 缺点 |
|---|---|---|
| **SSE**（Server-Sent Events） | 浏览器原生 EventSource API | Electron main 是 Node，没有 EventSource；格式啰嗦（`data: ...\n\n`） |
| **WebSocket** | 双向、低延迟 | 不需要双向，多一层协议握手 |
| **NDJSON**（newline-delimited JSON） | 最简——每行一个 JSON；用 fetch + ReadableStream 就能读 | 没有自动重连（但我们不需要） |

选 **NDJSON**：

- 每行一个 JSON：`{"type":"text","content":"Hi!"}\n{"type":"tool_use",...}\n`。
- `Content-Type: application/x-ndjson`。
- Electron main 和 browser 都能用 `fetch(url).then(r => r.body.getReader())` 读流，解码后按行 split + JSON.parse。
- 比 SSE 少一层抽象，比 WebSocket 少一层握手。

### 4.4 为什么 Backend 接口返回 `AsyncIterable`

```typescript
interface Backend {
  readonly provider: string;  // 'claude' | 'openclaw' | 'hermes' | 'pi' | ...
  execute(prompt: string, opts: ExecOptions): AsyncIterable<Message>;
}
```

候选还有三种：

- **事件回调**（`onMessage`）：丑，错误传播难。
- **Promise + 数组**（等所有消息攒完一起返回）：丢掉流式，UX 差。
- **ReadableStream**：Node 原生，但 `for await` 用 `AsyncIterable` 更顺。

`AsyncIterable` 的理由：

- **天然流式**：`for await (const msg of backend.execute(...))` 一条条处理。
- **取消干净**：调用方传 `AbortSignal`，backend 内部 spawn 的子进程跟随 abort。
- **错误传播**：backend 在迭代里 `throw`，调用方 try/catch 自然接住。
- **未来扩展**：openclaw / hermes / pi 的 stream 协议不同，但都映射到 `AsyncIterable<Message>` 即可——`Message` 是统一事件类型。

## 5. 与 multica 的架构对照

| 架构要素 | multica | 本项目 | 取舍理由 |
|---|---|---|---|
| 进程模型 | daemon 独立 | daemon 独立 | ✅ 沿用——崩溃隔离价值最大 |
| 健康端口 | `:19514 /health` | `:19514 /health` | ✅ 沿用 |
| `/shutdown` | POST 触发 cancel | POST 触发 abort | ✅ 沿用 |
| AbortController | Go ctx 树 | TS AbortController 链 | ✅ 语义等价 |
| Server HTTP client | ✅ register / heartbeat / poll / claim / result | ❌ 不需要 | 单机无 server |
| Task HTTP server | ❌ task 从 server 来 | ✅ task 从 UI 来 | 反向：daemon 从「client」变「server」 |
| Backend 接口 | Go `pkg/agent/Backend` | TS `Backend` interface | ✅ 沿用 |
| ClaudeBackend | `pkg/agent/claude.go` | `apps/daemon/src/agent/claude.ts` | ✅ 沿用 stream-json 模式 |
| daemon-manager | `apps/desktop/src/main/daemon-manager.ts` | 同名 TS 文件 | ✅ 沿用 |
| register runtime | 到 server | 到本机 shell（仅显示） | 简化 |
| 错误分类 | 401/404-task/404-runtime | 仅 task 级别（cancel/fail/timeout） | 简化 |
| 心跳 / poll loop | ✅ | ❌ | 不需要 |
| GC / auto-update / profile | ✅ | ❌ | 全砍 |

**继承的精髓**：进程隔离 + 健康协议 + Backend 抽象。
**去掉的包袱**：云端协同的全部复杂度。

## 6. 关键概念词汇表

| 词 | 含义 |
|---|---|
| **daemon** | 本地常驻进程，`apps/daemon` 的产物。 |
| **Backend** | agent CLI 的统一抽象。阶段 4 实现 `ClaudeBackend`，后续接 openclaw/hermes/pi。 |
| **DaemonClient** | shell 注入给共享 UI 的通信接口。desktop 版本走 IPC，web 版本走浏览器 HTTP。 |
| **task** | shell 推给 daemon 的一次执行请求。daemon 内部生成 task_id、管理生命周期、流式推事件。 |
| **stream-json** | Claude Code 的 `--output-format stream-json` 输出格式：每行一个 JSON 帧（text/thinking/tool_use/tool_result/result）。 |
| **NDJSON** | Newline-Delimited JSON。daemon → Electron 的流式协议，每行一个 JSON 事件。 |
| **health port** | 19514（默认）。daemon 唯一暴露的本地端口，只绑 `127.0.0.1`。 |
| **preflight** | daemon 启动后、`ready=true` 之前的阶段：探测 claude、起 HTTP server。`/health` 返回 `"starting"`。 |
| **liveness** | 「daemon 进程在跑」——端口在听就算活。 |
| **readiness** | 「daemon 可以接活儿」——preflight 完成、ready=true。 |
| **daemon-manager** | Electron 主进程里管理 daemon 子进程的模块。对标 multica 同名文件。 |
| **引导安装** | 首次启动检测 `claude --version`，缺失就引导用户 `npm install -g @anthropic-ai/claude-code`。 |
