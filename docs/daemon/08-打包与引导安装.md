# 08 · 阶段 7 — 打包 + 引导安装

> 终极目标：打出来的 desktop 安装包**直接能用**——双击安装 → 启动 → 自动检测 claude → 缺失就引导安装 → 装好后写 prompt → 收到回复。
>
> 这是 daemon 系列的收尾阶段。前面 6 个阶段都在 dev 模式（`tsx` 跑源码、`pnpm dev`），本阶段解决「交付给最终用户」的最后一公里。

---

## 1. 为什么需要这层

dev 模式下：
- daemon = `pnpm -C apps/daemon start`（tsx 实时编译 .ts）。
- claude = 开发者机器上已装、PATH 里能找到。

但**最终用户的机器上**：
- 没有 pnpm / tsx / 源码——daemon 必须是一个**自包含的可执行文件**。
- 没有 Node 开发环境——daemon 产物要内联所有依赖。
- claude 可能没装——要**检测 + 引导**。
- Electron GUI 应用的 PATH 与终端不同——claude 装在 npm global（`%APPDATA%\npm`）时，GUI 进程可能找不到。

这四个问题就是阶段 7 要解决的。

---

## 2. 四个核心概念

### 2.1 daemon 打包成单文件（tsup）

daemon 是 workspace 包（.ts 源 + `@demo/core` 源 + pino/zod 等 npm 依赖）。要变成「拷到任何机器、`node daemon.js` 就能跑」的单文件，用 **tsup**（esbuild 封装）把所有依赖**打进一个 bundle**：

```
src/main.ts ──tsup──> dist/daemon.js （内联 @demo/core + pino + zod）
```

关键配置：`noExternal: [/.*/]`——强制 bundle 所有依赖（默认 tsup 会 external 掉 dependencies，那样 prod 还得带 node_modules，失去单文件意义）。

> 为什么不是 `tsx` 直接跑源码：最终用户没源码、没 tsx。为什么不是 `tsc`：tsc 只转译不打包，留下一堆相对 import 指向不存在的 node_modules。tsup bundle 成单文件最省心。

### 2.2 desktop 用 extraResources 把 daemon 塞进安装包

electron-builder 打包时，默认只 bundle desktop 自己的代码（main/preload/renderer）。daemon 是**独立产物**，要显式声明 `extraResources` 拷进去：

```
打包后资源布局：
demo.exe
resources/
  app.asar              ← desktop 代码
  daemon/daemon.js      ← extraResources 拷进来的 daemon 产物
```

`electron-builder.yml`:
```yaml
extraResources:
  - from: ../daemon/dist/daemon.js
    to: daemon/daemon.js
```

### 2.3 prod spawn：`node resources/daemon/daemon.js`

`daemon-manager.resolveSpawnCommand` 的 dev/prod 分叉（阶段 6 只实现了 dev）：

| 模式 | 判定 | spawn 命令 |
|---|---|---|
| dev | `!app.isPackaged` | `pnpm -C apps/daemon start`（tsx 跑源码） |
| prod | `app.isPackaged` | `node <resources>/daemon/daemon.js` |

prod 路径用 `process.resourcesPath`（electron-builder 把 extraResources 放这）。env 同样注入 `DEMO_DAEMON_HEALTH_PORT`。

### 2.4 claude 引导安装 + GUI PATH 问题

**检测**：spawn `claude --version`（复用 daemon 的 probeClaude 思路，但 desktop main 自己实现一份，因为 desktop main 不依赖 daemon 包）。

**GUI PATH 坑**（Windows 重点）：Electron 打包后，进程继承的 PATH 不含 npm global 目录（`%APPDATA%\npm`），所以 `claude`（npm global 装的）spawn 不到。解法：检测前先 `npm config get prefix` 或读 `%APPDATA%\npm`，把它加进 spawn 的 `env.PATH`。

**引导**：检测不到时，renderer 顶部 banner 提示「未检测到 claude，请运行 `npm install -g @anthropic-ai/claude-code`」。装好后用户点 banner「重新检测」。

> 对照 multica：它 bundle 了整个 CLI（含 claude 下载管理）。我们按 `00-学习计划` 决策 8（4B）——**不 bundle claude**，claude 自己更新，我们只检测 + 引导。

### 2.5 claude 配置（provider + 认证）

装好 claude 不等于能用——尤其用**非官方 Anthropic** provider 时（如智谱 bigmodel），claude 还要配 `ANTHROPIC_BASE_URL` + token + 模型映射。这些都在 claude 自己的 `~/.claude/settings.json` 的 `env` 块里。

**设计：应用当 claude settings.json 的 GUI 编辑器**（不另搞一套 key 存储）：
- claude 自己读 `~/.claude/settings.json`（`env` 块它自动加载为进程环境），daemon spawn claude 时**不注入 env**——claude 用自己的配置，daemon 只 spawn。
- 应用读写该文件：显示路径 + 当前内容、提供「使用智谱（默认）」一键写入默认 env（用户只填 token）。

**智谱（bigmodel）默认模板**（应用内置，用户只填 `ANTHROPIC_AUTH_TOKEN`）：

| 字段 | 值 |
|---|---|
| `ANTHROPIC_BASE_URL` | `https://open.bigmodel.cn/api/anthropic` |
| `ANTHROPIC_AUTH_TOKEN` | `<用户填>` |
| `ANTHROPIC_MODEL` / `_SONNET/OPUS/FABLE_MODEL` | `glm-5.2` |
| `_HAIKU_MODEL` | `glm-4.7` |

> 注意：智谱用 `ANTHROPIC_AUTH_TOKEN`（**不是** `ANTHROPIC_API_KEY`）。写入时**合并**进现有 settings.json，保留用户已有的 `permissions` / `alwaysThinkingEnabled` 等。

**认证检测**：settings.json 的 env 有 token（`AUTH_TOKEN` 或 `API_KEY`）+ `BASE_URL` → 已认证。

**UI**：banner 显示 settings.json 路径 + 内容（可折叠）；未认证时提供 token 输入 +「使用智谱默认」按钮（合并写入）。改完即时生效（claude 下次跑读它，无需重启 daemon）。

> daemon 不持有凭证（决策 4 不变）：配置在 claude 的 settings.json，daemon 只 spawn claude。

## 4. 关键代码预演

### 4.1 daemon `tsup.config.ts`

```typescript
import { defineConfig } from "tsup";
export default defineConfig({
  entry: ["src/main.ts"],
  format: ["esm"],
  platform: "node",
  target: "node20",
  outDir: "dist",
  clean: true,
  sourcemap: true,
  noExternal: [/.*/], // 内联所有依赖（含 @demo/core 源码 + pino/zod），单文件可移植
});
```

### 4.2 `electron-builder.yml`（要点）

```yaml
appId: com.demo.desktop-web-demo
productName: demo
directories:
  output: release
files:
  - out/**/*
extraResources:
  - from: ../daemon/dist/daemon.js
    to: daemon/daemon.js
win:
  target: nsis
```

### 4.3 daemon-manager prod spawn

```typescript
private resolveSpawnCommand() {
  if (app.isPackaged) {
    const daemonJs = join(process.resourcesPath, "daemon", "daemon.js");
    return { cmd: process.execPath, args: [daemonJs], cwd: undefined };
  }
  // dev：见阶段 6
  const daemonDir = resolveDaemonDir();
  return { cmd: "pnpm", args: ["-C", daemonDir, "start"], cwd: daemonDir };
}
```

> 用 `process.execPath`（electron 自带的 Node）跑 daemon.js，不依赖系统 Node。

## 5. 验证

- **daemon 打包**：`pnpm -C apps/daemon build` → 产出 `dist/daemon.js`；`node dist/daemon.js` 能起、`curl /health` 返回 running（验证单文件自包含）。
- **desktop 编译**：`pnpm -C apps/desktop typecheck` + `electron-vite build`。
- **electron-builder 打包**：`pnpm -C apps/desktop build:desktop` → `release/` 产出（至少 `win-unpacked/resources/daemon/daemon.js` 存在）。完整安装包装机测试**需干净 Windows 机器**，本环境标注为人工。
- **claude 引导**：模拟 claude 不在 PATH（改 env）→ banner 显示。

## 6. 常见陷阱

1. **tsup 没打包 @demo/core**：默认 external 掉 workspace deps → daemon.js 还 import `@demo/core`，prod 找不到。解：`noExternal: [/.*/]`。
2. **GUI PATH 找不到 claude**：Electron 打包后 PATH 不含 npm global。解：claude-installer 探测 + 注入 PATH。
3. **extraResources 路径错**：`from` 是构建时相对路径，`to` 是安装包内路径；prod 用 `process.resourcesPath` 拼 `to`。
4. **pino 打包报错**：pino 极少数情况下有动态 require；若 tsup bundle 报错，单独 `external: ["pino"]` 并在 extraResources 带上 pino（少见，先试 bundle）。
5. **Windows spawn 弹黑窗**：daemon spawn 已 `windowsHide: true`；claude 检测 spawn 也要加。
