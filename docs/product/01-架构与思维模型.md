# 01 · 架构与思维模型（阶段 0）

> 本文是动手前的理论基础。读完它，你应当能回答：「我们这个工作台到底是什么？它和 multica、和 open-design 各是什么关系？为什么是 local-first + artifact-first？为什么把业务逻辑抽成共享内核、还支持双 runtime？这些选择又如何决定了后面所有的设计？」

---

## 1. 我们做的是什么——不是 multica，也不是 open-design

我们处在三个产品形态的交叉点上，但**不是其中任何一个**：

| | multica | open-design | **本项目** |
|---|---|---|---|
| **产品形态** | cloud-orchestrated 多机 agent 网络 | local-first **设计**工作台 | local-first **通用** agent 工作台 |
| **工作组织** | linear issue/task 流水线：web 建 issue → server 派发 → daemon 执行 → 回传 | brief → skill → design-system → artifact → handoff | folder → 选 agent+skill → 干活 → artifact → 多轮迭代 |
| **agent 在哪跑** | 某台机器的 daemon（server 派发） | 本地 daemon spawn 用户 CLI | **本地 shell 或云端 shell**（用户选） |
| **artifact** | task 结果（回传 server） | 设计产物（网页/PPT/dashboard/图片/动效） | 通用产物（代码/文档/原型/配置/脚本） |
| **数据主权** | server Postgres（中枢） | 本地 plain files | **本地 SQLite（本地模式）/ 云端 Postgres（云端模式）** |
| **渲染层** | Next.js（web） | Next.js + Electron | **Vite + TanStack + Electron**（已有） |

**我们向两者各借一半**：

- 从 **multica** 借**架构骨架**——「daemon + adapter：本地 daemon 是唯一特权进程，spawn agent CLI」。**本项目已有这套骨架**。open-design 也是这么做的（它的 References 明写「daemon + adapter … from multica」）。
- 从 **open-design** 借**产品形态**——「artifact-first 工作台：agent 产出可预览/可迭代/可导出的产物，用 skills 组合能力」。

**但我们的定位是「通用干活 + 双 runtime」，不只是它们**：

- 不走 multica 的 linear issue 模式：没有 server 派发、没有 issue→task 流水线。用户在工作台**直接**发起任务。
- 不限定设计领域：artifact 是任意「干活的产物」。`DESIGN.md` 只是可选的项目上下文，不是主线。
- **支持双 runtime**：agent runtime 可本地（桌面 daemon，web 也能连）可云端（server，像 multica 那样替用户跑）。用户自行选择——这正是 multica（云端）和 open-design（本地）两种形态的**并集**。

一句话：**multica 给了骨架，open-design 给了形态，我们做成「通用、能干活、local-first、且本地/云端双 runtime」的那个产品。**

## 2. 两个根本判断：为什么 local-first，为什么 artifact-first

整个工作台的设计，都是下面两个判断的推论。

### 2.1 为什么 local-first（数据与 agent 都在本地，不上云）

「local-first」不是「不能联网」，而是「**数据和执行的主权在用户的机器上**」。agent 联网调模型 API 是必然的（claude 要连 Anthropic），但**用户的项目、产出的 artifact、会话历史，默认都在本地**。

四条理由：

1. **隐私**：用户让 agent 干活的对象是**他自己的项目代码 / 文档 / 数据**。传到云端 server 是不必要的暴露面。local-first 让敏感内容不出本机。
2. **离线可用**：飞机上、内网里、断网时，只要 agent CLI 的模型可达（或本地模型），工作台照常干活。
3. **用户掌控**：数据是**用户的文件 / 本地 SQLite**——可以 `git add`、备份、迁移、用任意编辑器打开。不是锁在某个 app 的远端账号下。
4. **借力已有生态**：用户本来就装了 `claude`/`codex`，本来就在本机 `claude login` 存了 token。本地跑 agent 是顺势而为。

> local-first 不排斥云端——我们支持**云端 runtime 模式**（用户主动选），也支持**未来云端同步**。区别是：云端是**可选增强**，不是**前提依赖**。本地模式必须独立可用。

### 2.2 为什么 artifact-first（agent 产出「产物」，而不是「对话」）

「能干活」的本质，不是「agent 能聊天」，而是「**agent 能产出我可以直接用的东西**」。

| | chat 回复（现状） | artifact（目标） |
|---|---|---|
| 形态 | 一段 markdown 文字 | 结构化产物（HTML / 代码文件 / markdown 文档 / 图片） |
| 可预览 | ❌ | ✅（iframe / 高亮 / 渲染 / 图） |
| 可迭代 | ❌（只能重新生成） | ✅（局部修改、多轮精修） |
| 可导出/交付 | ❌（复制粘贴） | ✅（HTML 单文件 / ZIP / 就在项目里） |
| 可版本化 | ❌（刷新就丢） | ✅（文件 + SQLite 元数据，可 git） |
| 是「工作单元」 | ❌ | ✅（session 围绕 artifact 迭代） |

现状的本项目就是 chat-only 的反面教材：`tool_result` 被截断塞进 markdown，`messages` 是本地 `useState`，刷新即丢。**这就是我们要告别的。**

artifact-first 的推论：产物**落盘 + 元数据进 SQLite**、产物**可预览**（sandboxed）、session **围绕 artifact** 多轮、agent **在用户项目里**干活。

## 3. 数据流：一次「干活」的完整旅程

用户 import 文件夹、选 agent + skill、写任务，到看到 artifact 实时预览、多轮迭代。**下图是本地 runtime 模式**（本期主做）；云端 runtime 模式的差异在末尾说明。

```
┌─ renderer（desktop / web，共享 packages/views）──────────────────┐
│  ① 用户 import 本地文件夹 → project                              │
│  ② 选 agent（claude）+ skill（code-feature）                     │
│  ③ 写任务：「给这个 React 组件加暗色模式开关」                   │
│  ↓ runtimeClient.createSession({ projectId, agent, skillId })   │
│  ↓ runtimeClient.runTurn(sessionId, prompt)   // 订阅 NDJSON 流  │
└──────────────────────────────────────────────────────────────────┘
                ↓ HTTPS / localhost（NDJSON，复用现有 append-only 回放）
┌─ 本地 runtime shell = apps/daemon（演进）────────────────────────┐
│  Hono 路由 → packages/runtime 共享内核：                         │
│   session.manager:                                               │
│   - 解析 project（baseDir）+ skill（SKILL.md）                  │
│   - backendRegistry 取 session.agent 对应的 Backend             │
│   - backend.execute(prompt, {                                   │
│       cwd: project.dir,           // ← agent 在用户项目里干活    │
│       system: skill.contents,     // ← 注入 skill                │
│       signal, model, ...                                         │
│     })                                                          │
│   - 每条 Message → append 到 session 事件流 + 推订阅者           │
│   - artifact 事件 → 元数据写本地 SQLite + 内容写 project 文件   │
└──────────────────────────────────────────────────────────────────┘
                ↓ spawn + cwd = project dir
┌─ agent CLI（机器上已装）─────────────────────────────────────────┐
│  $ claude -p --output-format stream-json ...                    │
│   读项目文件 → 改代码 → 产出 artifact（写回项目目录）           │
└──────────────────────────────────────────────────────────────────┘

renderer 收事件：text/tool → 任务面板；artifact 事件 → debounce 100ms → sandboxed iframe 重建预览
④ 用户在预览看到结果 → 续接 turn「把开关默认设为开」→ 同 session 局部修改 artifact
```

**云端 runtime 模式的差异**（未来）：① ② ③ 不变，只是 `runtimeClient` 的 URL 指向云端 server（而非 localhost）；agent 在云端 server 跑（spawn server 端 CLI 或调 API）；数据写云端 Postgres，经 **Electric** 同步回本地（客户端用 TanStack DB `electric-collection` shape 订阅）。**业务逻辑（`packages/runtime`）完全相同**——这是共享内核的价值。

四个层次的关键设计：

- **renderer 只认 `RuntimeClient`**：runtime URL 可配（localhost / 云端）。preview 是普通 React 组件。
- **共享业务内核 `packages/runtime`**：session/artifact/skill/project 领域逻辑 + Drizzle schema + Hono 路由定义，**runtime 无关**。本地 shell 和云端 shell 复用同一份。
- **runtime shell 只差「agent 在哪跑 + 数据存哪」**：本地 shell = spawn 本机 CLI + 本地 SQLite；云端 shell = 云端 agent + Postgres + Electric。
- **NDJSON 协议不变，内容升级**：仍是 append-only 流式，事件从扁平 `TaskEvent` 升级为带 `sessionId`/`artifactId` 维度的 session/artifact 事件。

## 4. 关键设计决策

### 4.1 为什么共享业务内核 + 双 runtime（而非单 daemon 塞满，或重写）

这是本支线最核心的架构判断。功能增多后有三条路：

| 选项 | 评价 |
|---|---|
| 全塞进现有 `apps/daemon`（单进程、业务与 runtime 耦合） | ❌ 云端 runtime 要重写一遍业务逻辑 |
| 拆成本地 daemon + 本地 data-service 两个进程 | ❌ 多一层 IPC/部署，对单机工具过度 |
| **共享业务内核 `packages/runtime` + 两个 runtime shell**（本项目） | ✅ 业务一份、本地/云端各套一层 |

选**共享内核 + 双 shell**：

- 把**领域逻辑**（session/artifact/skill/project）+ **Drizzle schema** + **Hono 路由定义**抽到 `packages/runtime`，**runtime 无关**（不依赖 spawn、不依赖具体存储 dialect）。
- **本地 runtime shell**（`apps/daemon` 演进）：套上内核 + `@hono/node-server` + 本地 SQLite + 本地 agent adapter（spawn CLI）。**演进而非重写**——`Backend`/`Message`/NDJSON/`daemon-manager` 全复用。
- **云端 runtime shell**（未来 `apps/server`）：套上同一内核 + Hono + Postgres + Electric + 云端 agent + 认证。

agent 崩溃的隔离需求已由 spawn 子进程满足（`killProcessTree`），不需要靠拆进程。**共享内核让「双 runtime」和「未来云端」都不伤筋动骨**——这正是「方便未来 server 开发」的关键。

### 4.2 为什么混合存储（SQLite 元数据 + 文件内容），而非 plain files 或全 SQLite

| 选项 | 评价 |
|---|---|
| 纯 plain files + jsonl（open-design） | ❌ 无关系/查询/事务；无法做云端 sync |
| 全 SQLite（artifact 内容也入库 blob） | ❌ 大内容塞库不可读 diff、sync 笨重 |
| **混合：元数据 SQLite + 内容文件**（本项目） | ✅ 关系/查询/sync + git-reviewable 内容 |

- **元数据/关系/索引**（project/session/artifact meta/skill 注册/历史）→ **SQLite + Drizzle**：需要关系、查询、事务，且是 sync 的对象。
- **artifact 大内容**（HTML/代码/图片本体）→ **文件**（写 project 目录）：git-reviewable、可外部编辑、不塞 blob、契合「产物即文件」。

SQLite 存「artifact 元数据 + 文件路径」，内容是文件。open-design 全 plain files 是因为它数据简单且**不做 sync**；我们要关系 + 未来 ElectricSQL 同步，结构化存储更合适。

### 4.3 为什么 Hono 作共享 API 层（替换裸 node:http）

- 现在 `apps/daemon/src/health/server.ts` 用裸 `node:http` 手写路由，功能增多后（session/artifact/skill/project REST + SSE）会失控。
- **Hono runtime-agnostic**：同一套路由定义既能跑本地 daemon（`@hono/node-server`），又能跑未来云端 server——**双 runtime 共享 API 的基础**。
- Hono 的路由/中间件/zod body 校验/SSE helper 大幅简化，且与已暂停的 server 支线栈统一。
- 代价：给 daemon 加 Hono 依赖（极轻，值得）。

### 4.4 为什么 artifact-first（session/artifact 模型），而非保持 task 模型

现状 `task` 是一次性 prompt→执行，`sessionId` 只是透传字符串。升级为 `Session`（多轮，绑定 project+agent+skill+主 artifact）→ `Turn`（复用 `Message` 流）→ `Artifact`（结构化产物）。`Message`/NDJSON/append-only 回放全复用，外面包 session/artifact 维度。**移除旧 `/task/*`**（不留兼容）。

### 4.5 为什么 sandboxed iframe（`srcdoc`、无 allow-same-origin）

artifact（agent 生成的 HTML/JS）是**不可信代码**。`<iframe sandbox srcdoc>`（无 allow-same-origin）让 artifact 在隔离 origin 跑，摸不到 host 的 `window`/cookie/DOM。直接 `dangerouslySetInnerHTML` = XSS 灾难；带 allow-same-origin = 隔离失效。流式更新靠 debounce 100ms 重建 `srcdoc`。

### 4.6 为什么继承 agent 自己的权限模型，而非发明自己的沙箱

agent 在用户真实项目里干活（读写文件、跑命令）是危险动作。但 Claude Code 本来就有 `--allowed-tools`/`--permission-mode`，codex 有 sandbox——**已经很成熟**。daemon 只设**边界**（cwd 限定 project dir + `realpath`/`resolveSafe` 防越界），**执行级权限交给 agent 自己**。不重复造轮子、不给用户双重权限心智（对齐 open-design 安全模型）。

### 4.7 为什么 Vite + TanStack + Electron（含 TanStack DB），而非照搬 Next.js

open-design 用 Next.js 为 SSR + Vercel serverless + 一键部署，**这三条我们都不需要**。而我们已有成熟的 Vite + TanStack + Electron 共享渲染层，换 Next.js = 推翻 frontend 成果。

**额外好处**：TanStack 全家桶让我们顺理成章接入 **TanStack DB**——本地模式用它的 **SQLite persistence**（本地 SQLite 主、离线），云端模式用 **`electric-collection`**（Postgres 主 + Electric shape sync）。客户端数据访问统一在 TanStack DB collection，只是 backend 换。代价（云端模式需自建 server shell，无 Vercel direct）我们接受。

## 5. 与 multica / open-design 的架构对照

| 架构要素 | multica | open-design | 本项目 | 取舍 |
|---|---|---|---|---|
| 进程模型 | daemon 独立 | daemon 独立（唯一特权进程） | 本地 shell + 共享内核 | ✅ 沿用 + 抽内核 |
| 业务逻辑位置 | 与 server runtime 耦合 | 与 daemon 耦合 | **`packages/runtime` 共享内核** | ✅ 新增（双 runtime 复用） |
| runtime 位置 | 仅云端（server） | 仅本地（daemon） | **本地 + 云端（用户选）** | ✅ 取两者并集 |
| agent 引擎 | daemon 自己跑 | spawn 用户 CLI（22+） | spawn 用户 CLI | ✅ 沿用 |
| API 层 | chi（Go） | Express | **Hono**（本地+云端共享） | ✅ 新选型 |
| 工作组织 | linear issue 流水线 | artifact-first 循环 | artifact-first 循环（通用） | ✅ 取 open-design |
| 数据存储 | Postgres | plain files + jsonl | **SQLite+Drizzle（元数据）+ 文件（内容）** | ⚠️ 混合（为 sync） |
| 数据主权 | server Postgres | 本地文件 | **本地 SQLite / 云端 Postgres（按模式）** | ✅ 双模式 |
| 客户端数据访问 | — | daemon API | **TanStack DB**（SQLite persistence / electric-collection） | ✅ 新增 |
| 未来云端 sync | 内置（中枢） | ❌ | **ElectricSQL（Postgres-centric）** | ✅ 预留 |
| 预览 | 无 | sandboxed iframe + 流式 | sandboxed iframe + 流式 | ✅ 取 open-design |
| skills | 无 | `SKILL.md`（259+） | `SKILL.md`（通用，少量） | ✅ 精简版 |
| 权限模型 | server 管 | 继承 agent 自己的 | 继承 agent 自己的 | ✅ 取 open-design |
| 渲染层 | Next.js | Next.js + Electron | **Vite + TanStack + Electron** | ⚠️ 有意差异 |
| 多机协同 | ✅ | ❌ | ❌ | ✂️ 砍 |
| sidecar / BYOK / MCP | — | ✅ | ❌（延伸） | ✂️ 砍 |

**继承的精髓**：daemon+adapter 骨架（multica）+ artifact-first 形态（open-design）+ sandboxed preview + 继承 agent 权限。
**超越两者**：**共享业务内核 + 双 runtime**（本地/云端可切）+ TanStack DB 统一数据访问 + ElectricSQL 未来同步。
**砍掉**：multica 的 issue 流水线 + 多机；open-design 的设计向重度 / sidecar / BYOK / MCP / 多导出格式。

## 6. 关键概念词汇表

| 词 | 含义 |
|---|---|
| **工作台 / Studio** | 本支线的产品形态：local-first 通用 agent 工作台。 |
| **local-first** | 数据与执行的主权在用户机器上；本地模式独立可用（离线），云端是可选增强。 |
| **artifact-first** | agent 产出结构化「产物」（而非对话）；产物可预览/迭代/导出/版本化。 |
| **共享业务内核（`packages/runtime`）** | runtime 无关的领域逻辑（session/artifact/skill/project）+ Drizzle schema + Hono 路由定义。本地 shell 与云端 shell 复用同一份。 |
| **runtime shell** | 套上共享内核的运行时外壳。**本地 shell**（`apps/daemon` 演进）= 本地 SQLite + spawn 本机 CLI；**云端 shell**（未来 `apps/server`）= Postgres + Electric + 云端 agent + 认证。 |
| **Runtime / RuntimeClient** | agent runtime 的抽象 + 客户端接口。URL 可配（localhost / 云端），上层不感知 runtime 位置。 |
| **runtime 模式 ↔ 数据主权** | 选本地 runtime → 本地 SQLite 主；选云端 runtime → 云端 Postgres 主 + Electric 同步。 |
| **artifact** | agent 产出的结构化产物（html/code/markdown/image）。内容写文件，元数据进 SQLite。 |
| **project** | 用户的一个工作单元 = 一个本地文件夹（folder import 或新建）。agent 在其中干活。 |
| **session / turn** | project 内的多轮会话 / 其中一轮（复用 `Message` 流）。 |
| **Backend / adapter** | agent CLI 的统一抽象（`execute → AsyncIterable<Message>`，已有）。 |
| **backendRegistry** | agent 名 → Backend 工厂的注册表；session 按 `agent` 路由（阶段 3）。 |
| **skill** | 一个 `SKILL.md`，决定 agent 怎么干活。可扫描、热重载、注入。 |
| **sync-friendly schema** | 为未来同步设计的 schema 约定：UUID 主键、`created_at`/`updated_at`、软删除 `deleted_at`、`user_id`/`device_id`。 |
| **ElectricSQL（Postgres-centric）** | 未来云端同步方案：云端 Postgres 主 + Electric sync engine + TanStack DB `electric-collection`（shape 订阅、乐观写、`txid` 匹配）。 |
| **TanStack DB** | 客户端 reactive 数据访问层。本地模式用 SQLite persistence，云端模式用 electric-collection——backend 换、上层不变。 |
| **sandboxed preview** | `<iframe sandbox srcdoc>`（无 allow-same-origin）预览 artifact，隔离不可信代码。 |
| **继承 agent 权限模型** | daemon 只设 cwd 边界，执行级权限交给 agent 自己（`--allowed-tools` 等），不发明自己的沙箱。 |
