# 06 路 阶段 5: 平台能力抽象

> 目标：在“路由层同构”的前提下，把 web 和 desktop 永远不可能相同的系统能力隔离出来。
> 本阶段只做一条最小链路：共享页面点击按钮后，web 走 `window.open`，desktop 走 `preload + IPC + shell.openExternal`。

---

## 1. 这一章解决的不是路由问题，而是平台问题

阶段 4 已经证明：

- 同一个 `HomeView` 可以同时跑在浏览器和 Electron 窗口里
- `@demo/views` 可以直接使用 TanStack Router
- 不需要 `NavigationAdapter`

但这不代表所有代码都能直接共享。

一旦共享页面想做下面这些事，就会触到平台边界：

- 打开外链
- 发原生通知
- 读写文件
- 调系统对话框
- 访问剪贴板

这些能力在 web 和 desktop 的实现天生不同，所以共享页面不能直接写：

- `window.open(...)`
- `window.desktopAPI.xxx(...)`
- `new Notification(...)`
- Electron 的 `dialog`、`shell`、`clipboard`

结论只有一个：

> 路由层可以直接共享；平台能力必须抽象后再注入。

---

## 2. 这一章的结构重点：先分清“脚手架已有部分”和“项目新增部分”

从阶段 4 开始，`apps/desktop` 已经刻意收敛到更接近官方 `react-ts` 脚手架的结构。
因此阶段 5 的正确讲法，不是“我们从零造了一套 Electron 通道”，而是：

1. 官方脚手架已经给了我们 `main` / `preload` / `renderer` 三层外壳
2. 我们在这个外壳之上，新增了一条“平台能力注入”链路

所以本章要明确区分两类代码：

### 官方脚手架层

这些东西本来就属于 Electron 外壳：

- `apps/desktop/src/main/index.ts`
- `apps/desktop/src/preload/index.ts`
- `apps/desktop/src/preload/index.d.ts`
- `apps/desktop/src/renderer/index.html`
- `apps/desktop/src/renderer/src/main.tsx`

### 本项目在脚手架上的增量层

这些才是本教程真正新增的部分：

- `packages/core/platform/types.ts`
- `packages/core/platform/context.tsx`
- `apps/web/src/platform/capabilities.ts`
- `apps/desktop/src/renderer/src/platform/capabilities.ts`
- `packages/views/home/home-view.tsx`
- `apps/web` / `apps/desktop` 对 Provider 的注入

这个分层一定要记住，因为它决定了后续维护时该去哪里找问题：

- Electron 壳问题，先对照官方模板
- 共享能力边界问题，先看 `@demo/core`

## 3. 本阶段的最小目标

这一章只定义一个能力：

```ts
export interface PlatformCapabilities {
  openExternal(url: string): Promise<void> | void
}
```

为什么只做一个？

因为“打开外链”已经足够穿透完整链路：

1. 共享页面触发动作
2. `@demo/core` 提供统一接口
3. web 与 desktop 各自实现
4. desktop renderer 调 preload
5. preload 经 IPC 调主进程
6. 主进程调用系统能力

这条链路一旦走通，通知、文件、剪贴板都只是同一模式的扩展。

---

## 4. `@demo/core` 里到底放什么

本阶段新增的核心包是 `@demo/core`。

它不负责 UI，也不负责平台具体实现，只负责两件事：

1. 定义能力接口
2. 提供依赖注入的 Context 管线

关键文件：

| 文件 | 作用 |
|---|---|
| `packages/core/platform/types.ts` | 定义 `PlatformCapabilities` 接口 |
| `packages/core/platform/context.tsx` | 定义 `PlatformCapabilitiesProvider` 和 `usePlatformCapabilities()` |

这里最重要的边界是：

- `@demo/core` 不碰 Electron API
- `@demo/core` 不碰浏览器专属 API
- `@demo/core` 不写业务页面

它只是中间那层抽象。

## 5. web 端实现：最薄的一层

web 端实现放在：

[`apps/web/src/platform/capabilities.ts`](D:/Projects/Frontend_tech/electron/electron-tanstack-router-lessons/apps/web/src/platform/capabilities.ts)

逻辑非常简单：

```ts
export const webPlatformCapabilities = {
  openExternal(url) {
    window.open(url, '_blank', 'noopener,noreferrer')
  },
}
```

这里没有任何复杂度，因为浏览器原生就有这项能力。

## 6. desktop 端实现：建立在官方脚手架外壳之上

desktop 端和 web 最大的不同是：renderer 不能直接碰系统能力。

所以 desktop 端需要分成三层：

### 第 1 层：renderer 中的项目实现

文件：

[`apps/desktop/src/renderer/src/platform/capabilities.ts`](D:/Projects/Frontend_tech/electron/electron-tanstack-router-lessons/apps/desktop/src/renderer/src/platform/capabilities.ts)

它只是把共享接口转发给 preload 暴露出的 API：

```ts
export const desktopPlatformCapabilities = {
  async openExternal(url) {
    await window.desktopAPI.openExternal(url)
  },
}
```

### 第 2 层：preload 白名单桥

文件：

[`apps/desktop/src/preload/index.ts`](D:/Projects/Frontend_tech/electron/electron-tanstack-router-lessons/apps/desktop/src/preload/index.ts)

这部分本质上还是官方脚手架的 preload 壳，只是我们在上面增加了自己的 `desktopAPI`：

```ts
const api = {
  platform: 'desktop' as const,
  openExternal(url: string) {
    return ipcRenderer.invoke('platform:open-external', url)
  },
}
```

然后再通过 `contextBridge.exposeInMainWorld('desktopAPI', api)` 暴露给 renderer。

也就是说：

- 官方模板负责 preload 这层的安全形态
- 本项目只是在这个形态里塞入自己的白名单能力

### 第 3 层：main 进程系统能力

文件：

[`apps/desktop/src/main/index.ts`](D:/Projects/Frontend_tech/electron/electron-tanstack-router-lessons/apps/desktop/src/main/index.ts)

这里通过：

```ts
ipcMain.handle('platform:open-external', (_, url: string) => {
  return shell.openExternal(url)
})
```

把请求落到真正的系统能力 `shell.openExternal()`。

这才是 desktop 端最终执行动作的地方。

## 7. Provider 注入：平台实现如何进入共享页面

共享页面不应该自己判断“我现在是在 web 还是 desktop”。

正确做法是由 app 层注入：

- web 根路由注入 `webPlatformCapabilities`
- desktop 根路由注入 `desktopPlatformCapabilities`

这样 `@demo/views` 中的页面只需要：

```tsx
const { openExternal } = usePlatformCapabilities()
```

然后直接调用：

```tsx
<Button onClick={() => openExternal('https://github.com/multica-ai/multica')}>
  打开 multica 仓库
</Button>
```

页面无需知道：

- 当前是浏览器还是 Electron
- 底层是 `window.open` 还是 `shell.openExternal`
- 中间有没有 preload 和 IPC

这就是依赖注入带来的稳定边界。

## 8. 共享页面这一层应该保持多干净

本阶段之后，`packages/views/home/home-view.tsx` 的职责应该非常纯：

- 组合共享 UI
- 通过 `@demo/core` 拿能力接口
- 触发业务动作

它不应该直接出现这些内容：

- `window.open`
- `window.desktopAPI`
- `ipcRenderer`
- `shell`
- Electron 类型

这条纪律比“按钮能不能点开链接”更重要。
因为这决定了页面是否真的还能被 web 与 desktop 长期共用。

---

## 9. 为什么这里用 Context 是合理的

本项目约定是：

- 服务端状态给 TanStack Query
- 客户端业务状态给 Zustand
- Context 只用于平台管线

这一章就是“平台管线”的标准例子。

`PlatformCapabilitiesProvider` 不是业务状态容器，而是依赖注入容器。
它承载的是“当前外壳能提供什么能力”，不是“当前业务状态是什么”。

所以：

- 这里用 Context 合理
- 如果把筛选条件、抽屉开关、当前工作区放进同一个 Context，就不合理

## 12. 小结

阶段 5 真正补上的不是某一个按钮，而是一条共享架构纪律：

> Electron 官方脚手架负责给出安全的 `main / preload / renderer` 外壳；本项目在这个外壳之上，通过 `@demo/core` 定义平台能力接口，再由 app 层分别实现并注入给共享页面。

所以这一章最重要的不是 API 名字，而是边界意识：

- 外壳尽量靠近官方模板
- 能力接口放进 `@demo/core`
- 平台实现留在 app 层
- 共享页面只依赖抽象
