# 07 路 阶段 6: 复刻主要 UI

> 目标：开始把 multica 风格的主界面壳真正做出来。
> 这一阶段不追求一次性做完整业务，而是先把最关键的三层结构搭对：`DashboardLayout`、`AppSidebar`、`PageHeader`。

---

## 1. 这一章先复刻什么，不复刻什么

参考 `D:\Projects\src\multica` 后，最应该先学的不是它庞大的 issues 业务逻辑，而是它的主界面分层：

1. `DashboardLayout`
2. `AppSidebar`
3. `PageHeader`

这三层决定了整个产品的空间组织方式。

本阶段先复刻：

- 左侧工作区侧栏
- 顶部固定页头
- 内容区的卡片式 dashboard 骨架
- web / desktop 共享的主界面壳

本阶段暂不复刻：

- 完整的 issues / projects 查询逻辑
- 复杂筛选器
- 右侧详情栏
- 拖拽、排序、批量操作
- 多页面完整业务流

原因很简单：

> 学架构时，先把“壳”搭对，比过早堆业务细节更重要。

## 4. 对齐 multica 的地方在哪里

### 4.1 总布局分层

我们现在也采用这条结构：

```txt
SidebarProvider
├── AppSidebar
└── SidebarInset
    └── Page content
```

这和 multica 的主界面心智是一致的。

### 4.2 Sidebar 的信息架构

目前侧栏也采用了与 multica 接近的三段式：

- 侧栏头部：工作区块 + 主要动作
- 中部导航：个人 / 工作区 / 配置 三组
- 底部辅助区：当前用户/环境信息

虽然现在还是静态内容，但信息架构已经和后续真实产品扩展方向一致。

### 4.3 PageHeader 的密度

页头保持：

- 高度固定 `h-12`
- 细边框分隔
- 文本层级克制
- 移动端显示侧栏触发器

这也是 multica 主界面的重要风格特征之一。

---

## 5. 刻意没有照搬的地方

### 5.1 不照搬业务复杂度

multica 的 `AppSidebar` 很重，里面有：

- 工作区切换
- 邀请处理
- runtime 更新提示
- pinned items
- DnD 排序
- 快捷键
- query/mutation 接线

本项目这一阶段刻意不做这些。

原因：

- 这些会把教程重点从“布局分层”拉偏到“业务堆叠”
- 学习项目要先复刻结构，再逐步加行为

### 5.2 不照搬路由抽象

multica 因为 web 和 desktop 路由不同，需要 `NavigationAdapter`。
本项目继续保持：

- 共享页面直接使用 TanStack Router
- 不额外造导航抽象

### 5.3 不把平台能力混进 sidebar

本阶段侧栏虽然是主界面的一部分，但“打开外链”这类动作依然必须走 `@demo/core`。
也就是说，主界面壳升级了，平台边界纪律不变。

## 6. 当前结果应该长什么样

完成后你应该能看到：

- 左侧有固定侧栏
- 侧栏有工作区块、三组导航、底部信息区
- 右侧顶部有固定页头

- web 和 desktop 视觉结构保持一致

这说明我们已经从“能共用页面”走到了“能共用主界面骨架”。

---

## 7. 为什么先做静态骨架是对的

很多人会在这一步急着开始：

- 接真实数据
- 做复杂筛选
- 写多个页面
- 做右侧详情抽屉

但如果侧栏、页头、内容区层级都还没稳定，这些东西很快就会反复返工。

所以本阶段的策略是：

1. 先把 dashboard 的空间组织稳定下来
2. 再把页面一个个挂进去
3. 再往每个页面里填真实交互

这是更接近真实产品开发的顺序。

## 9. 小结

阶段 6 的核心收获不是“已经做完 multica UI”，而是：

> 先把 multica 风格主界面的布局骨架抽成共享壳，再逐步把真实页面和交互挂上去。

换句话说，这一步复刻的是产品的“空间结构”，不是产品的“全部业务”。

下一步可以继续往两个方向推进：

1. 把事项 / 项目 / 设置等页面骨架真正挂成多路由
2. 把顶部 header 做成更接近 multica 的过滤 / 显示控制区
