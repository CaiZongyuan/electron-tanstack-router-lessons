# 01 · 架构与思维模型（阶段 0）

> 本文是动手前的理论基础。读完它，你应当能回答：「我们这个 server 到底在干什么？它和 multica 的 server 有什么本质差异？为什么认证是一个独立远端服务，而不是塞进 daemon 或 Electron 主进程？」

---

## 1. 我们要做的不是 multica server

multica 的 server 是一个 **cloud-orchestrated 的 task 中枢**：用户在 web 创建 issue → server 派发给某台机器的 daemon → daemon 跑 `claude`/`codex`/… → 结果回传 → server 经 realtime WebSocket 推给前端。认证、计费、调度、agent 执行、实时推送……全挤在一个 Go 服务里，认证只占边缘一小块。

**我们要做的是「远端认证边缘服务」**：把 multica server 里那一小块认证边缘裁下来，用 TypeScript 重写，砍掉整条 task 管线。它对外只有三件事——**登录、license、设置同步**，**不碰 task、不碰 agent、不碰 daemon**。

| | multica server | 本项目 server |
|---|---|---|
| **角色** | task 中枢 + agent 调度 + 认证边缘（全都要） | 只做认证边缘 |
| **语言/栈** | Go + chi + Postgres/sqlc + Redis | TS + Hono + SQLite→Postgres + Drizzle |
| **Task 分发** | 核心：issue → daemon claim → agent → 回传 | ❌ 不做（AGENTS.md 第 6 节） |
| **认证方式** | 邮箱验证码（无密码）+ JWT + HttpOnly cookie + CSRF + `mul_` PAT | 密码 + argon2id + opaque session token + Bearer |
| **License** | ❌ 无此子系统 | ✅ Ed25519 签名（自设计） |
| **设置同步** | `/api/me`（user 表上的列）+ `notification_preference` | `/settings`（用户级 KV 表） |
| **与 daemon 关系** | server 派 task 给 daemon，daemon 回传 | ✗ 完全无关系（两条平行线） |
| **实时推送** | realtime WebSocket Hub + Redis Streams fanout | ❌ 不做 |
| **部署形态** | 多节点 + Redis relay | 单节点容器起步 |

**为什么还要对标 multica 的 server**？因为它继承了 multica server 认证边缘的三条精髓：

1. **「一个用户身份、多端共享」的中心化心智**（密码/session/license 都在远端集中保管）。
2. **token 鉴权 + `/me` 设置同步**这套标准 API 形态（multica 的 `internal/handler/auth.go` + `pkg/db/queries/user.sql`）。
3. **中间件链**：鉴权 / 限流 / 日志 / CORS（multica 的 `internal/middleware/`）。

学完这套骨架，你既理解了 multica server 认证那一块的设计，也做出了一个有用的远端认证服务。未来若想接真实 multica server 或自建 cloud，骨架已经就位——只是再把 task 管线补回来。

## 2. 为什么认证是独立远端服务，而不是塞进 daemon 或 Electron 主进程？

这是整个架构里**最关键的判断**。一旦理解它，后面所有设计（远端部署、Bearer token、secure storage、`AuthClient` 抽象）都是它的推论。

「用户登录」这件事，理论上有三个候选落点：

### 2.1 候选 A：塞进本地 daemon —— ❌

daemon 已经是本机 agent runtime。能不能让它顺手管登录？

不能。三条理由：

- **daemon 刻意不持有凭证**（`daemon/00` 决策 4）：claude CLI 自己管 token，daemon 是纯 transport。把用户密码 hash 塞进去，违背这条干净边界。
- **认证天然是中心化的**：一个账户要能从 desktop、web、未来更多端登录。本地 daemon 存的东西换台机器就没了。
- **职责混淆**：daemon 的核心价值是「跑 agent + 流式输出」。认证是另一个领域（identity），混进去会让两个子系统互相拖累。

### 2.2 候选 B：塞进 Electron 主进程 —— ❌

Electron 主进程能不能直接管用户账户？

不能。两条理由：

- **主进程是本机的**：用户的「账户」不能存在某台机器的 Electron 内存/磁盘里——换机器、重装系统就丢了。账户必须是远端的。
- **web 壳子没有 Electron 主进程**：`apps/web` 跑在浏览器，它要登录就必须有一个**远端可访问**的认证端点。把认证塞进 Electron，web 端就用不了。

> Electron 主进程的真正职责是**持有 token 并代理请求**（token 存 secure storage，renderer 经 IPC 让它代发 HTTP），而不是**判定身份**。判定身份永远在远端 server。

### 2.3 候选 C：独立远端服务 —— ✅

认证做成独立远端服务，四条理由：

1. **凭证集中**：密码 hash、session、license 私钥**只**在 server。客户端不持有任何可伪造身份的东西（token 是 server 签发的凭证，不是身份本身）。
2. **跨设备**：desktop、web、未来 CLI/移动端，共享同一个身份源。一次注册，处处登录。
3. **与 agent 执行解耦**：认证 server 挂了，**不影响本地 daemon 跑 task**（daemon 不需要 server）；反过来 daemon 崩了，不影响用户登录。两个崩溃域彻底隔离。
4. **可独立部署/升级**：认证服务可以单独迭代（换 hash 算法、加 OAuth、改 session 策略）、单独扩容、单独打安全补丁，不动 daemon 也不动前端。

### 2.4 三条独立的「崩溃域」与「部署域」

```
┌─ 远端 ──────────────────────────────────────────────┐
│  apps/server（独立机器/容器）                         │
│   持有：密码 hash、session、license 私钥、用户设置    │
│   暴露：HTTPS · /auth/* /license/* /settings/*        │
└──────────────────────────────────────────────────────┘
            ▲ Bearer token over HTTPS          ▲
            │                                   │
┌─ 本机：apps/desktop ────────────────┐   ┌─ apps/web ─────────┐
│  Electron 主进程                     │   │  浏览器 renderer    │
│   持有：token（secure storage）       │   │  AuthClient(fetch)  │
│   职责：代理请求、spawn/管 daemon     │   └─────────────────────┘
│        │ spawn + localhost            │
│        ▼                              │
│  apps/daemon（独立进程）              │   ← daemon 与 server 完全无关
│   持有：无凭证                         │
│   暴露：127.0.0.1:19514 /task/*       │
└───────────────────────────────────────┘
```

三个圆各自独立挂掉都不波及其他：server 挂 → 本机 daemon 照跑 task、只是登不了录；daemon 崩 → 用户照常登录同步设置、只是跑不了 agent；Electron 卡死 → server 和 daemon 都不受影响。

## 3. 数据流：一次登录与一次设置同步的完整旅程

共享的 `packages/views` 登录页只认一个 `AuthClient` 接口。用户输入邮箱密码到看到自己的用户信息，shell 侧有两条接线，但下游 server API 完全一致：

```
┌─ packages/views 共享登录页 ───────────────────────────┐
│  用户输入 email/password，点登录                       │
│  ↓ authClient.login({ email, password })             │
└───────────────────────────────────────────────────────┘
                ↓
┌─ desktop 壳子 ─────────────────────────────────────────┐
│  renderer                                             │
│   ↓ IPC: 'auth:login'                                 │
│  preload                                              │
│   ↓                                                   │
│  main / AuthClient                                    │
│   fetch('https://<server>/auth/login', POST, body)    │
│   ← { token }                                          │
│   secureStorage.set('auth_token', token)              │
└───────────────────────────────────────────────────────┘
                ↓ 或者
┌─ web 壳子 ─────────────────────────────────────────────┐
│  browser AuthClient                                   │
│   fetch('https://<server>/auth/login', POST, body)    │
│   ← { token }（或 server Set-Cookie: httpOnly）        │
│   token 存 sessionStorage / 由 cookie 自动带          │
└───────────────────────────────────────────────────────┘
                ↓ HTTPS · POST /auth/login
┌─ server ──────────────────────────────────────────────┐
│  /auth/login handler:                                 │
│   - 查 user by email                                  │
│   - argon2id.verify(user.passwordHash, password)      │
│   - 失败 → 401（统一文案，不泄漏「邮箱是否存在」）       │
│   - 成功 → 生成 opaque session token（随机 bytes）      │
│   - sessions 表插入 SHA-256(token) + user_id + 过期    │
│   - 返回 { token, user }（token 只此一次明文出现）      │
└───────────────────────────────────────────────────────┘

之后每次请求（如 GET /me、PUT /settings）：
  client 在 header 带：Authorization: Bearer <token>
  server 中间件：SHA-256(token) → 查 sessions 表 → 注入 ctx.user
```

设置同步的方向反过来是「client 发、server 存、另一端拉」：

```
desktop PUT /settings { theme: 'dark' }
  → server 写 user_settings (user_id, 'theme', '"dark"', updated_at=now)
web    GET /settings
  → server 读该用户所有 KV → { theme: 'dark', ... }
```

三个层次的关键设计：

- **共享视图只看 `AuthClient`**：`packages/views` 的登录页/设置页不知道 IPC，也不写死 server URL。这样同一套 UI 才能同时跑在 desktop 与 web（与 `daemon/01` 的 `DaemonClient` 同构）。
- **desktop main 是 token 保险箱 + 代理**：token 进 Electron `safeStorage`（OS 级加密），renderer 永远摸不到明文 token，只能经 IPC 让 main 代发请求。
- **web adapter 直连同一套 HTTPS API**：web 不需要 main 进程，浏览器直接 `fetch`；token 走 httpOnly cookie 或 storage。
- **server 是身份唯一裁判**：从 HTTPS 进，到 DB 出，密码永远不出 server、token永远不存明文。

## 4. 关键设计决策

### 4.1 为什么密码登录，而不是 multica 的邮箱验证码

multica 的登录是 **magic-link / 邮箱验证码**（`internal/handler/auth.go` 的 `SendCode` + `VerifyCode`，6 位码进 `verification_code` 表，无密码列）。

候选还有：

| 选项 | 优点 | 缺点 |
|---|---|---|
| **密码 + argon2id**（本项目起步） | 零外部依赖、本地可跑、教学闭环完整 | 需自己管 hash/重置流程 |
| **邮箱验证码**（multica） | 无密码、防撞库 | 需邮件发送服务（SMTP/Resend），起步依赖重 |
| **OAuth/SSO**（multica 有 Google） | 体验好、少管密码 | 需注册 OAuth app、配 client secret，最重 |

选**密码起步**的理由：

1. **最窄可跑**：学习项目第一目标是「跑通认证闭环」，密码方案不需要任何外部服务，`curl` 就能验。
2. **能讲全安全细节**：hash 算法、salt、session、限流、撤销——密码方案把这些都逼出来，教学价值最大。
3. **可演进**：auth 子系统设计成「凭据策略可插拔」，后续加验证码/OAuth 只是多一个 login 路径，不动 session/中间件。

> multica 选验证码是因为它是面向公众的产品，要降低注册摩擦、避免密码泄露责任。我们是学习项目，优先教学闭环。

### 4.2 为什么 opaque session token，而不是 JWT

multica 用 **JWT（HS256）+ HttpOnly cookie**（`internal/auth/jwt.go`）：无状态、server 不存 session、靠签名验真。

候选对比：

| 选项 | 优点 | 缺点 |
|---|---|---|
| **opaque session token**（本项目） | 可撤销（删 sessions 表一行即失效）、DB 只存 hash、教学能讲 session 表 | 每次请求查一次 DB |
| **JWT**（multica） | 无状态、不查 DB | 撤销难（要黑名单/短 TTL+refresh）、payload 暴露（只签名不加密） |

选 **opaque token**：

1. **client 是非浏览器（desktop app）+ 浏览器**：multica 用 JWT+cookie 是因为它纯面向浏览器、追求无状态。我们的 desktop app 用 cookie 别扭，Bearer header 更通用；既然要带 token，opaque 和 JWT 一样能用。
2. **可撤销是刚需**：「登出」「改密码后全端下线」「license 异常踢下线」都依赖「能立刻让 token 失效」。opaque token 删一行就失效；JWT 要维护黑名单，反而更复杂。
3. **教学闭环**：opaque 方案逼你设计 `sessions` 表（token hash、user_id、过期、撤销位），把「会话是什么」讲透。

> 「JWT 无状态省一次 DB 查询」在认证场景几乎不是瓶颈（session 查询走索引，亚毫秒）。可撤销性的价值远大于省这一次查询。

### 4.3 为什么 Drizzle ORM，而不是 Prisma / sqlc

multica 用 **sqlc**（写 SQL → 生成 Go 类型化查询，`sqlc.yaml` + `pkg/db/queries/*.sql` → `pkg/db/generated/`）。TS 生态里对应的几个选项：

| 选项 | 优点 | 缺点 |
|---|---|---|
| **Drizzle**（本项目） | 类型安全、查询即 SQL（无新 DSL）、迁移可控、运行时极轻、dialect 可切 | 生态比 Prisma 小 |
| **Prisma** | 生态大、工具全 | 运行时重（query engine 二进制）、生成客户端、查询是自有 DSL、远离 SQL |
| **Kysely** | 纯查询构建器、类型好 | 不管迁移、不管 schema 定义 |

选 **Drizzle**：

1. **最贴近 multica 的 sqlc 心智**：写 SQL 风格的查询、schema 即代码、生成类型。学完能直接对照 multica 的 sqlc 理解「为什么这么分」。
2. **轻**：纯 TS、无二进制运行时，部署镜像小（阶段 8 受益）。
3. **dialect 可切**：起步 `better-sqlite3`，生产换 `postgres-js` —— schema 几乎不动，只换 driver（阶段 3/8 展开）。

### 4.4 为什么 Ed25519 离线 license

multica **没有 license 子系统**（全仓 grep 只在 skill 文件名过滤命中 "license"）。所以这是本项目的**自我设计**。关键选择：

- **非对称签名（Ed25519）而非对称（HMAC）**：server 持私钥签发，client 持**公钥**验签。client 离线也能验（不需要联网问 server「这个 license 有效吗」），且 client 即使被逆向后也**无法伪造** license（没有私钥）。对称签名的话 client 一泄露密钥就能无限伪造。
- **离线验签而非查表校验**：desktop 应用离线时也要能用（飞机上、内网）。查表校验把可用性绑死在 server 在线。
- **Ed25519 而非 RSA**：签名短（64 字节）、快、参数简单（没有「选哪个 padding / 多长的模数」的坑）、现代首选。

payload 设计（`packages/core/server/license.ts`）：`{ userId, email, plan, features[], expiresAt, issuedAt }` + Ed25519 签名。client 验签 + 验过期 + 验用户匹配，三步都离线。

### 4.5 为什么 `AuthClient` 抽象（与 `DaemonClient` 同构）

`packages/views` 的登录页/设置页要在 desktop 与 web 都跑。两种壳子连 server 的方式不同（desktop 经 IPC 走 main、web 直接 fetch），但**语义完全一致**（login/logout/me/getSettings/putSettings）。

于是复用 `daemon/01` 的同款套路：在 `@demo/core` 定义传输无关的 `AuthClient` 接口，两端各注入实现。共享 UI 只依赖接口，不感知传输——和 `DaemonClient` 一模一样，只是协议从「localhost NDJSON」换成「远端 HTTPS JSON」。

> 一句话：**daemon 那条线（`DaemonClient`）和 server 这条线（`AuthClient`）是同一个抽象模式的两次应用**。学会一个，另一个就是抄。

## 5. 与 multica server 的架构对照

| 架构要素 | multica | 本项目 | 取舍理由 |
|---|---|---|---|
| 语言 | Go | TypeScript | ✅ 换 TS（与前端/daemon 同构，一份语言栈） |
| Web 框架 | chi（标准库 net/http） | Hono | ✅ 类比——都是「轻路由 + 中间件」哲学 |
| 数据库 | Postgres | SQLite（dev）→ Postgres（prod） | ✅ 起步用 SQLite 零依赖，生产对齐 |
| 查询层 | sqlc（生成 Go） | Drizzle ORM | ✅ 类比——都是「写 SQL + 生成类型」 |
| 缓存/限流 | Redis | 内存（起步） | 简化——单节点不需要 Redis |
| 登录方式 | 邮箱验证码 | 密码 argon2id | 简化——起步最窄可跑（见 4.1） |
| 会话 | JWT + HttpOnly cookie + CSRF | opaque session token + Bearer | 调整——非浏览器 client + 可撤销（见 4.2） |
| PAT（`mul_`） | ✅ `/api/tokens` CRUD | 可选延伸 | 简化——先不做，session 够用 |
| OAuth | ✅ Google | ❌ 起步不做 | 简化——延伸项 |
| 用户表 | `001_init` user + 多个偏好列 migration | `users` 表 + `user_settings` KV | ✅ 沿用心智，KV 更通用 |
| 设置同步 | `/api/me` PATCH（user 列） | `/settings` GET/PUT（KV） | ✅ 沿用心智，KV 表解耦 |
| License | ❌ 无 | ✅ Ed25519 签名 | **新增**（multica 无对应，自设计） |
| 中间件链 | auth/ratelimit/csp/client/logger | auth/ratelimit/logger/cors | ✅ 沿用，砍掉 workspace/cloudfront |
| Task 分发 | ✅ `/api/daemon/*` claim/start/complete… | ❌ 不做 | ✂️ 砍（AGENTS.md 第 6 节） |
| daemon 编排 | ✅ `internal/daemon` + `pkg/agent` | ❌ 不做 | ✂️ 砍（daemon 在本机自治） |
| WebSocket 唤醒 | ✅ `daemonws` | ❌ 不做 | ✂️ 砍 |
| 定时任务 | ✅ `scheduler`（sys_cron） | ❌ 不做 | ✂️ 砍 |
| realtime 推送 | ✅ `realtime` + `events` | ❌ 不做 | ✂️ 砍 |
| 多机协同 | ✅ | ❌ | ✂️ 砍（单机工具） |

**继承的精髓**：token 鉴权 + `/me` 设置同步 + 中间件链 + 「一个身份多端共享」的中心化心智。
**去掉的包袱**：整条 cloud task 管线 + realtime + 多机协同。
**新增的**：license（Ed25519 离线验签）。

## 6. 关键概念词汇表

| 词 | 含义 |
|---|---|
| **server** | 本系列的产物，`apps/server`，远端 Hono + TS 认证服务。 |
| **AuthClient** | shell 注入给共享 UI 的认证通信接口。desktop 版走 IPC over main，web 版走浏览器 fetch。与 `DaemonClient` 同构。 |
| **opaque session token** | 不透明会话令牌：server 生成的随机 bytes，DB 只存其 SHA-256 hash。client 持明文 token，每次请求带 Bearer header；server 哈希后查表。可撤销。 |
| **argon2id** | OWASP 当前首选的密码 hash 算法（内存硬化，抗 GPU/ASIC 爆破）。本项目用 `@node-rs/argon2`。 |
| **secure storage** | Electron 的 `safeStorage`（或 keytar）：把 token 用 OS 级加密存盘（Windows DPAPI / macOS Keychain）。renderer 摸不到明文。 |
| **license** | server 用 Ed25519 私钥签发的凭证 token，绑定用户/到期/特性。client 用公钥**离线验签**。multica 无此物。 |
| **Ed25519** | 现代非对称签名算法（Edwards 曲线）。签名 64 字节、快、无参数坑。server 私钥签、client 公钥验。 |
| **last-write-wins** | 设置同步的初始冲突策略：并发写以 `updated_at` 最新者为准。简单够用，更复杂的（版本号/CRDT）作为延伸。 |
| **Bearer token** | HTTP 鉴权方式：`Authorization: Bearer <token>`。本项目会话与 license 都走它（license 还多一层签名）。 |
| **崩溃域** | 一个能独立故障而不波及其他的单元。server / daemon / Electron 主进程是三个独立崩溃域。 |
| **drizzle-kit** | Drizzle 的迁移工具：`generate` 从 schema diff 生成 SQL migration，`migrate` 执行。 |

## 7. 阶段 0 的「完成标志」

读完本文，你应该能对以下问题给出简短回答（自测即可）：
